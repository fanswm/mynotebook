Masterway 来了
1. 一点「小小的」不满
了解一件事物最好的办法，就是看它是怎么来的。
所以，在正式跟大家介绍 Masterway 之前，先聊聊痛苦的过往吧。

和很多人一样， 一直有做笔记， 写日记，记任务和写文章的习惯。用过很多很棒的 App 来分别完成这些事情。但有个问题，一直没有好的办法解决，就是 信息分散得很严重，这里一堆，那里一块，杂乱无章。每当需要再次整理和查看时就得在多个地方来回折腾，效率着实低下。信息日积月累，效率低便慢慢变成痛苦了。

举个例子吧，如果想通过某条笔记找到要做的任务，就得切换出笔记 App 打开任务 App 再找任务；稍稍复杂一点的信息联想，那就得在多个 App 之间来回切换了，而且这还是在你能够记得住这些信息在这些 App 内的理想情况下。据我的经验，在这个信息堆积如山的现代社会里，过不了三五天，基本上就很难搞清楚了。

信息零散，不便管理，效率低下，这还没说不同 App 各有各的操作流程，各有各的需要掌握的基本操作知识。一句话，够折腾人。

老实讲，如果可以，这些折腾的时间，我更愿意拿来刷刷剧，撸撸猫，陪陪家人。

其实回头看看，不难发现，无论是随笔，日记还是工作任务，文档等等信息，要么是与自己直接相关的，譬如学习，生活和工作的事情，要么是自己与自己间接相关（当然，也可能是直接相关的），如关注的，感兴趣的事物等等。

因为都与自己有关，所以它们（信息）之间天生就是有联系的。

那么，能不能充分利用这种联系，使我们即简简单单，又不失强大地集中管理这些信息呢？

这便是 Masterway 诞生的理由。

不是有某某某了吗？嗯，我想我们该 give it a try，and make it better！

2. 一个地方
One place to store them all, 把他们集中在一起，免除信息分散带来的烦恼。

信息很分散，怎么办？那么就把信息集中在一起吧。别看简简单单的一句话，解决起来可并不容易！信息之所以分散，在于信息本身多种多样，各有各的特点。正是因为这个原因，有很多不同的 App 会专注于处理某些细分的应用场景 。其实这进一步造成了信息的分散化，进而带来不必要的烦恼；

老实讲，没有一种完美的方式能够完美处理所有的信息。尽管如此，我们仍希望通过创造和努力，从信息本身的因素出发，慢慢向这个目标靠近，方便自己，造福大家。

2.1 集中存储

所谓，乱麻怕快刀，奇迹自大力。首先，不论信息的形式，笔记，任务，文章还是别的形式的信息，先集中存放在一起，从最根本的层面解决信息分散的问题。这看似粗暴，实则是解决信息分散化，提高信息管理效率最最基础，最不可缺的一步。

目前有的信息存储方式，大致可以分为以下几种：

一种是完全自己动手，搭建存储方案，同时自己提供信息访问功能。这种方案对用户有一定的技术能力要求，同时也需要用户投入时间和精力来进行维护，以及进行新功能的开发；

一种是 App 只提供信息访问能力，而存储需要用户自己购买并配置第三方存储方案；这种方案，同上一种方案相比，降低了对技术能力的依赖，但还是需要一些技术知识作为支撑才会有比较好的体验。

一种是 App 自主提供依托于操作系统的存储方案。如 Apple 系，使用iCloud作为存储端，用来在 Apple 系的产品上访问信息；由于依托于操作系统能力，这种方案会让 App 在访问自由度上受到一定程度的限制，譬如 Web 访问，不同操作系统环境下（如 Android 系）的访问。

还有一种是 App 提供包括存储在内的一站式服务。如 Evernote、SimpleNote 等等。让用户免去一切有关存储的烦恼。用户只需要关注的信息本身，同时信息访问能力也不会受到上述几种方案固有的限制。就目前而言，这种做法可以算是存储中最方便的方式了。

Masterway 采用了最后一种方式，将所有信息集中存储在云端，同时提供多端访问的能力，这样，无论是在桌面端，还是在网页上，都可以随时随地记录，整理和查看信息。

2.2 历史变更记录

当然，Masterway 能做的，不仅仅是将信息集中在一起。为了彻底免除另一个大的，信息变更 的问题，Masterway提供了历史版本的服务。简单讲，你的每一次编辑都会被贴心的保存起来，方便你回头随时回顾。

3 一个规则
One rule to rule them all, 标记信息，合理隔离，从而更好的管理他们

信息集中存储是解决信息分散非常有效的办法，但仅仅存储在一起是远远不够的，信息集中化势必造成信息形式多样，信息量巨大。该如何方便地整理，如何方便地查看都是需要解决的问题。

答案还得从信息自身出发去查找。我们知道，任何事物都是抽象特征具体化的产物，信息亦是如此。信息在多个维度有很多自己的特征，譬如信息类型，是任务，是日记，是摘要还是完整的文章? 譬如信息范畴，是关于学习，关于生活还是工作的？ 譬如信息具体的关注点，是关于管个人兴趣，UI设计，产品调研，还是关于市场分析的等等。无论信息内容多么复杂，形式多么多样，只要抓住了信息的特征，那么这条信息的内容就可以通过特征来进行表示了。

在信息整理中，有两大非常重要的归类方法，目录归档法（又称文件夹等，后统称目录）和标记法（后称标签）。二者各有优缺，具体不在此深究，简单讲，目录具有强关联特性，适合隔离信息的场景，标签具有松散特性且是信息自身的子信息，适合完整表述信息特征，聚集信息。

如果能够提取信息的特征，并充分利用上述两大归类法，是不是能够解决掉信息集中化带来的一部分问题呢？我们看看 Masterway 是怎么做的。

3.1 标记它

在 Masterway 中，一条信息称为一条 Line（这样叫是为了有一个简单的名称来抽象所以的信息，譬如是日记，笔记，任务等等，总之可以 Masterway 中 Line 即信息，信息也叫 Line），在记录过程中，用户可以根据信息的特征，结合自身需要，自由地给Line打上足够多的标签。就像这样。

null
如你所见，Masterway 会自动补齐你曾经用过的标签，让这一个原本看起来“很复杂”的过程变得非常简单，大大提高了标记的效率。

就打标签而言，Masterway 是非常鼓励这样的行为的，出于两点考虑：
一，对于一条信息而言，如果标签越多越详细，说明信息被作者剖析的越详尽，作者也越容易对信息有更加深入的理解；
二，信息标签越多，Masterway 会自动形成更多的信息链接，提高多条信息之间的关联性，从而更好地帮助作者管理和记忆。

当然，享受标签带来的好处和便利的前提是标记的每一个标签都能够真实的反应信息本身的特征，而不是一些堆砌在一起的，与信息不相关的词汇。

3.2 放在 Board 中

有了标签之后，利用标签的松散特性，就可以非常容易的将有共同标签的 Line 聚合起来了。举个例子，如果有Line标有 task 的标签，那么，通过标签 task，我们就可以找到所有包含了这个标签的 Line 了。这其实是一个非常简单，同时又非常强大的功能。

3.3 使用 Project 来隔离它们

我们知道，一个人平时能够接触的信息实在是太多太多了，但也不能看出，其实一些信息之间是没有太多共性的。譬如，化学论文和流行音乐的共性就比较少，又譬如不同行业的专业知识之间的共性也是比较少的。在这样的情况下，如果对信息不做处理，还是一股脑儿的把它们仍在一起，那么，即使有 Board 和标签的神力加持，信息一多，不免也会无比杂乱，难以整理。为应对这样的情况，我们可以把目录的隔离特性充分地利用起来了。

Masterway 利用 项目（即目录）将信息从根本上进行隔离，让属于不同项目的 Lines 彼此独立；同时，独立带来的另一个好处是，只有属于同一项目的Line才会共享该项目的标签集合，这在一定程度上很好地规避了大量标签必然造成的标签意义混淆，难以管理的问题。

4. 一个网络
One web to link them all, 将他们（信息）连接起来，构建信息网络

5.2 也谈谈付费

老实讲，这是是一个比较敏感的话题。一来，历来习惯，在初次见面就谈付费的事多少有些不合时宜，二来，网络上就软件服务付费已经有很多很棒的谈论。所以，一而再的斟酌是否要谈这个话题，但既然是要真诚地说说心里话，那也就可以放下这些包袱了。

我想说两点。
一，付费永远是用户自主的事情。
就付费而言，是否要买服务和是否要买橘子本质上是一样的，永远是用户自己决定，而非商家能够支配的事情。

二，付费是价值交换最简单的方式，也是价值持续交换最稳固的方式。
用户付出一部分费用，购买一定价值的服务，满足自己的需要，这个过程是非常简单直观的。同时，价值提供方有了资金的支持，能够改善产品，稳定服务，持续的交付更加完善的服务给用户，从而形成良性循环，最终实现共赢，所以这是一条最稳固的方式。

对 Masterway 这类服务而言，这一条尤为重要的。Masterway 旨在给大家提供非常非常棒的服务，让你在学习，生活和工作中能够得到更好的提升，而付费是让这一愿望得以实现的最靠谱的方式。

好消息是，现在国内很多用户对付费已经有了更加全面的看法和认识，对 付费享受服务 的接受程度也越来越高，这着实是服务提供方的一个福音，其实换换角度，这又何尝不是用户的福音呢？
